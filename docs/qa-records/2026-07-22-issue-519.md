## Root cause and repair accounting

The alerted source was committed at `2026-07-21T21:09:28Z`. Its CI and Rust
runs were created at `21:09:32Z`. The first gate failure was sampled only 24
seconds after the commit and 20 seconds after workflow creation. The second
sample was 70 seconds after the commit and 66 seconds after workflow creation.
At both samples, the CI real-libmpv integration job and the Rust workspace job
were still running. CI completed successfully at `21:12:12Z`, and Rust
completed successfully at `21:11:55Z`.

Pull request #512 had already changed pending exact-SHA workflow evidence inside
a 900-second window from an immediate failure to a blocking warning with reason
`source-main-ci-settling`. It kept completed failures immediate and made overdue
pending evidence fail closed. However, its first implementation anchored the
window to commit creation time and did not accept a successful empty workflow
query as pending evidence. Those gaps could still misclassify a delayed push or
a newly pushed source whose workflow runs had not appeared in the API index.

Pull request #524 completed the repair by anchoring settling to the matching
GitHub `PushEvent`, falling back to matching workflow creation time, and
treating a successful empty workflow query as pending rather than an API
failure. The evaluator and live collector retain regression coverage for
settling, overdue pending evidence, completed failures, delayed pushes, empty
run histories, and missing event-feed evidence. After #524 landed, Maestro
recorded that the configured runtime outcome was verified in the closure of
[`#508`](https://github.com/BeFeast/ok-player/issues/508#issuecomment-5039482974).

The public evidence proves that #519 was not a completed source-workflow
failure. The external scheduler's private sample payload is unavailable, so it
cannot distinguish whether these two samples came from an older deployed
checkout or the valid-empty-run-index edge. The hardened source behavior and
the later configured-runtime verification cover both remaining paths without
weakening completed-failure or timeout handling.

The generated pipeline context named
`rust/crates/okp-core/src/playback_failure.rs`, but the complete exact-source
workflow evidence contains no playback diagnostic failure. Editing playback
failure classification would therefore be unrelated to the observed gate
defect.
