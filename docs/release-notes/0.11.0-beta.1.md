OK Player is a native GTK4 media player over libmpv, sharing the pure-Rust `okp-core` with the
Windows WinUI 3 line. When published, this will be the first build on the **beta** channel;
earlier Linux builds shipped on the alpha line (`0.1.0-linux-alpha.*`).

## User-visible changes

Playback and video:

- Native Wayland/EGL video plane with a retained `GtkGLArea` fallback, including validated 4K60
  HEVC Main10 presentation.
- Loaded video is fit inside the monitor work area, and the initial fit is consumed once per source
  rather than fighting later manual resizes.
- Reliable double-click-to-fullscreen on video, with the OSC and titlebar clearing at idle.
- Failed local media loads are retried; network URLs have explicit loading, live, and error states.

Subtitles:

- Curated subtitle style presets (size / position / style) with live preview and an OSC lift that
  keeps captions clear of the control band.
- Subtitle search with cue navigation, dual-subtitle support that keeps the secondary caption
  distinct, WebVTT sidecar passthrough, and labeling of image-based (PGS/VobSub) tracks.

Navigation and chapters:

- Live seek hover thumbnails on the timeline, timecode and frame-number seek readouts, and chapter
  support with an interval fallback plus **Detect chapters** for files without metadata.

Audio and lyrics:

- Floating volume control distinct from the audio-track switcher, an audio quick-switcher with
  language tags and audio delay, and a synced-lyrics surface for audio playback.

Windowing and shell:

- Compact mini-player mode, a player context menu reachable from non-interactive surfaces, Settings
  that stay visible during playback, and rarely-used video geometry controls (aspect, zoom/pan,
  rotation, crop, deinterlace) grouped under **Advanced commands → Video**.

Integration and identity:

- Companion resume and progress report-back seam, `ok-player://` URI scheme reservation, NFO sidecar
  titles, and the canonical vector application identity/icon set.

Capabilities deliberately reported as **Deferred** in this build: gapless playback, honest HDR
handling diagnostics, and A–B clip/GIF export. Each is surfaced in the UI as a deferred/unavailable
state rather than silently missing — see [Known limitations](#known-limitations).

## Install, update, rollback, and uninstall

**Debian / Ubuntu (`.deb`)**

```bash
sha256sum -c SHA256SUMS
sudo apt install ./ok-player_0.11.0-beta.1_amd64.deb
```

**AppImage**

```bash
chmod +x OK-Player-0.11.0-beta.1-x86_64.AppImage
./OK-Player-0.11.0-beta.1-x86_64.AppImage
```

**Update.** Both lanes poll the static HTTPS feed and apply updates in place; no command to run.

**Rollback.** Install an earlier `.deb` (`sudo apt install ./ok-player_<older>_amd64.deb`) or run an
earlier AppImage; downgrades are not blocked.

**Uninstall.** `sudo apt remove ok-player` for the `.deb`, or delete the AppImage file (and any
`.desktop` entry you added). Per-user JSON settings and history are left in place; remove them by
hand for a clean slate.

## Known limitations

- **Gapless playback — Deferred.** The GTK shell owns queue order and waits for libmpv's
  end-of-file before the next `loadfile`, so libmpv gapless-audio mode is not enabled; Settings →
  Playback shows this as a disabled **Deferred / Unavailable** row.
- **HDR — diagnostics only.** HDR handling is reported honestly as unimplemented rather than
  faked; there is no tone-mapping claim for this build.
- **A–B clip/GIF export — Deferred.** The A–B states are reserved in the UI but export is not wired.
- **Codec / GPU.** Hardware decode depends on a working VA-API stack; without it, playback falls
  back to software decode and 4K60 HEVC Main10 presentation is not guaranteed.
- **Compositor / Wayland.** Desktop-integration behaviors — native file/folder choosers, drag/drop,
  clipboard, portals, compositor fullscreen, keyboard focus — are validated only under
  GNOME/Wayland by the operator level. Other compositors may differ; those rows are not claimed
  passing outside a live operator run.

## Acceptance summary

Evidence is graded by the levels in [`docs/linux-release-acceptance.md`](../linux-release-acceptance.md):
model-unit, headless Xvfb render, installed-package, and live GNOME/Wayland operator. For this beta:

- **model-unit** and **headless Xvfb render** are reproduced by the release harness
  (`scripts/run-linux-acceptance-harness.sh`), which writes `xvfb-rows.json` and fails on any
  canonical redline miss.
- **installed-package** launch/version checks run against the candidate `.deb` and AppImage.
- **Live GNOME/Wayland operator rows** — file/folder chooser, drag/drop, clipboard, portal,
  compositor fullscreen, keyboard focus — remain `not-run` until the operator completes them on a
  real GNOME/Wayland session and merges the results with
  `scripts/merge-linux-acceptance-evidence.sh`. This release is **not** eligible to publish until
  those rows pass.

Owner of the live acceptance pass and the final `⟨…⟩` values above: the release operator. This
document must not be treated as acceptance evidence on its own — a headless run proves deterministic
rendering, not a real desktop flow.
