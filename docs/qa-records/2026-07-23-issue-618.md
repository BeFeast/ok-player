## Incident and repair accounting

False-green run `29990133877` was requested for
`a30b603a523f3df35822f04c0e5b9696ced56f7a`. It completed with GitHub
conclusion `success`, but its publication transaction reported
`stale_generation`: the requested and built SHA remained `a30b603`, current
`main` had advanced to `e6925a9`, generation `174` was allocated, build `169`
was still public, and the stale reason was `requested_head_changed`. The
workflow summary recorded `public feed: untouched`. The public pointer
therefore remained on source `89f92cb`; workflow success did not deliver the
new tip.

Recovery run `29991171359` was requested for exact source
`e6925a977153e8cb83b87d6d0ae1feb3e16ea753`. It published generation `175`,
moving `candidate.linux.json` to that SHA at `2026-07-23T08:53:52Z`. The
accepted pointer and its Debian, AppImage, Velopack, and checksum identities
are bound below.

The initial implementation merged through PR #619 correctly introduced
`candidate-success-without-delivery` and
`candidate-repeated-success-without-delivery`, but Greptile demonstrated two
P1 gaps. First, it used a workflow trigger SHA to infer what a coalesced run
published, which could page even when the run produced the current feed.
Second, it limited a mixed run list before filtering, so active or unrelated
events could displace completed green non-deliveries.

Tested source `08882d5cc356b02b062bc537f3c82a2deb51a5be` repairs both findings.
The collector queries completed `schedule` and `workflow_dispatch` histories
separately, merges the newest 100, and queries active `in_progress` and
`queued` runs separately. Every candidate transaction now emits one compact
`OKP_CANDIDATE_OUTCOME_JSON` record. For recent successful runs while the feed
is behind, the collector carries the publish result and built SHA from the
complete Actions log into the core snapshot. The evaluator suppresses recovery
only when `publish_result=published` and the built SHA equals the public feed;
`stale_generation` and `skipped` remain non-delivery. Missing or contradictory
transaction evidence fails closed as `candidate-run-history-unavailable`.

The final live checker at the rebased tested source exited `0`. Source CI was
still inside its separate 15-minute settling window, and the Linux row passed
because accepted build `175` was 2,055 seconds behind the first unpublished
commit, inside the 7,200-second delivery bound. The deterministic collector
regression separately classifies an exact-main `in_progress` run at 7 seconds
as `candidate-delivery-in-progress`, while the evaluator regression rejects the
same evidence after the 5,400-second run limit.
