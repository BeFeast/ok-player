## Root cause and repair accounting

The last Windows Candidate run before issue #594 was run `29971043226`. It
started at `2026-07-23T01:09:01Z`, passed checkout, C# build, unit tests,
real-libmpv integration tests, package assembly, identity validation, head
revalidation, and feed-last promotion, and completed successfully at
`01:13:31Z`. The rolling manifest therefore validly identified build `31` from
source `fc119fb59f9eaece2b7bcab456cc802a01da6dc7`; there was no Windows builder
or product gate failure to repair.

`main` first advanced beyond that source at `01:27:37Z` and continued through
`de333e7e24ff77c45362d7258febbdb13d00d79b`, but GitHub created no later
scheduled Windows Candidate run before the issue was filed at `03:55:57Z`.
The live versioned checker reproduced the issue at `04:07:43Z`: source CI and
the static Windows feed passed, while `windows-candidate-delivery` failed
because the first-unpublished-commit lag reached 9,606 seconds and exceeded
the 7,200-second contract. The complete sanitized JSON is preserved in the
issue comment linked below.

The workflow previously depended on GitHub's best-effort scheduled event as
its only automatic trigger. A `*/15` cron expression does not guarantee that
GitHub will create a run every 15 minutes, so schedule delay could exhaust the
delivery SLA even while the builder and published candidate remained valid.

Repair PR #596 added a `push` trigger scoped to `main` and retained the schedule
as a fallback. Push and scheduled runs both refresh `origin/main` immediately
after checkout and exit successfully before SDK setup when their requested SHA
has already been superseded. This makes a burst of merges coalesce to the newest
source without spending hosted Windows build time on queued stale commits.
Manual dispatch deliberately bypasses the early supersession decision and
retains its existing explicit republish and diagnostic behavior.

Merging that repair produced push run `29979874356`. It passed every Windows
gate, promoted build `33` from exact source
`f612365f0014e9de58df8d52854fa2d6a3e1a4f8`, and completed at `04:38:33Z`.
At `04:43:42Z`, the live checker reported `windows-candidate-delivery=pass` and
confirmed that build `33` exactly matched current `main` at that sample.

PR #599 extended failure-streak monitoring to the new primary push trigger, but
its bounded query fetched up to 100 runs before excluding manual dispatches.
Greptile's authoritative review demonstrated that 100 newer manual runs could
therefore displace all automatic history and reset a real push/schedule failure
streak to zero. Tested source
`03b98a24a0501330ac582a4df72723a3556a7f90` queries completed push and scheduled
runs separately, merges the newest 100 automatic results, and derives both the
consecutive-failure count and newest failed gate from that automatic-only
history. The live-collector regression supplies three consecutive failed push
runs, an older successful scheduled run, and a full 100-run manual-dispatch
page that the collector must never request.

Greptile's authoritative review of PR #603 then found that the manual-saturation
fallback in that regression was not observable: the repaired collector always
supplied an event filter, so the test passed without asserting that the filters
were actually present. It also merged by completion/update time, which can
misorder overlapping runs when an older run finishes after a newer one.

Review-finding tested source `bb294446715d375675db318fea854eaeb13a8911`
records every fixture-backed GitHub CLI invocation and asserts that the Windows
history consists of exactly two bounded completed-run queries, one for `push`
and one for `schedule`, with no unfiltered request. Its push and schedule
fixtures are deliberately interleaved: three failures are newest by creation
time, while the older success has the newest update time. The collector now
sorts by creation time with run identity as a deterministic tie-breaker before
counting the streak, so both query filtering and newest-first merging are
reviewable regression contracts.

After those monitoring fixes merged, two additional hosted runs exposed
delivery-path failures that were not product or package failures. Push run
`29982204563` stopped while reading the existing `candidate.windows.json`
because the release-asset connection was reset. The next push run succeeded,
showing that the rolling pointers were valid and the transport interruption was
transient. The workflow nevertheless had only one attempt for each prior
pointer and therefore converted a recoverable read into a failed automatic run.

Push run `29984383316` then passed checkout, compilation, both Windows test
suites, packaging, and candidate identity assembly. Before upload, `main`
advanced from its requested source `27dd4d7e0ba671a92ccfc5e5500ff552c2af49a6`
to `6ec9c535b8ae31eda239696da3274c00b6c2f7f6`. The final head guard correctly
prevented publication but deliberately failed the job. Because project health
counts completed automatic failures, an expected late coalescing event could
therefore enter the `windows-candidate-builds-failing` streak even though no
builder gate or release mutation was defective.

Delivery-hardening tested source
`a167f3806057e2b641ab9322020b144f20c48cfa` gives both prior pointer downloads
three bounded attempts with short backoff. It also records a late supersession
as a successful skip before any release lookup or upload, and gates pruning on
an actual successful promotion. The focused regression pins the retry bound,
the skip-before-mutation ordering, the observable supersession token, and the
pruning guard. Successor run `29984534115` promoted build `38` from exact current
`main`; the live checker then exited `0` with
`windows-candidate-delivery=pass`.

The generated pipeline context named
`rust/crates/okp-core/src/playback_failure.rs` and
`rust/crates/okp-core/src/player_commands.rs`, but the complete successful
builder log and live health detail contain no playback diagnostic or player
command failure. Editing those portable domains would be unrelated to the
observed delivery-trigger defect.

## Limitations and holds

- The post-merge push trigger and exact-main candidate promotion are proven.
  Fail-safe handling of repeated push failures and manual-run saturation is
  fixture-backed because deliberately breaking the public Windows delivery
  lane would be destructive.
- Build `38` and the live checker prove that the existing lane recovered and
  exactly matches current `main`. The new retry and successful late-supersession
  branches are regression-backed before merge; deliberately forcing either
  condition against the public rolling release would be destructive.
- Rolling manifest and feed URLs intentionally advance. The exact build-33
  bytes are preserved in the issue comment and bound by checksums; Full packages
  use versioned asset names and digests.
- Hosted Windows Actions prove compilation, unit behavior, real-libmpv
  integration behavior, package assembly, and rolling-feed identity. They do
  not prove installation, updater migration, GPU decode, HDR, 4K60, physical
  audio, multi-monitor behavior, or any live Windows desktop state.
- The full live checker also reported an unrelated Linux candidate failure.
  This repair neither changes nor claims acceptance of the Linux delivery lane.
- No workflow was manually dispatched by this investigation, no recovery
  command was invoked or replaced, and no release, package, feed, tag, or
  repository setting was mutated by this investigation.
