## Root cause and repair accounting

The last Windows Candidate run before issue #594 was run `29971043226`. It
started at `2026-07-23T01:09:01Z`, passed checkout, C# build, unit tests,
real-libmpv integration tests, package assembly, identity validation, head
revalidation, and feed-last promotion, and completed successfully at
`01:13:31Z`. The rolling manifest therefore validly identified build `31` from
source `fc119fb59f9eaece2b7bcab456cc802a01da6dc7`; there was no Windows builder
or product gate failure to repair.

`main` first advanced beyond that source at `01:27:37Z` and continued through
`de333e7e24ff77c45362d7258febbdb13d00d79b`, but GitHub created no later
scheduled Windows Candidate run before the issue was filed at `03:55:57Z`.
The live versioned checker reproduced the issue at `04:07:43Z`: source CI and
the static Windows feed passed, while `windows-candidate-delivery` failed
because the first-unpublished-commit lag reached 9,606 seconds and exceeded
the 7,200-second contract. The complete sanitized JSON is preserved in the
issue comment linked below.

The workflow previously depended on GitHub's best-effort scheduled event as
its only automatic trigger. A `*/15` cron expression does not guarantee that
GitHub will create a run every 15 minutes, so schedule delay could exhaust the
delivery SLA even while the builder and published candidate remained valid.

Repair PR #596 added a `push` trigger scoped to `main` and retained the schedule
as a fallback. Push and scheduled runs both refresh `origin/main` immediately
after checkout and exit successfully before SDK setup when their requested SHA
has already been superseded. This makes a burst of merges coalesce to the newest
source without spending hosted Windows build time on queued stale commits.
Manual dispatch deliberately bypasses the early supersession decision and
retains its existing explicit republish and diagnostic behavior.

Merging that repair produced push run `29979874356`. It passed every Windows
gate, promoted build `33` from exact source
`f612365f0014e9de58df8d52854fa2d6a3e1a4f8`, and completed at `04:38:33Z`.
At `04:43:42Z`, the live checker reported `windows-candidate-delivery=pass` and
confirmed that build `33` exactly matched current `main`.

The Greptile review on PR #596 identified a remaining monitoring gap: the
collector queried completed Windows Candidate runs with `--event schedule`, so
repeated failures from the new primary push trigger could fall through to
generic publication-lag evidence. Tested source
`fcd00c77dbbe192e17307a70c15852101a64db3e` removes that event filter, selects
the combined push and scheduled automatic history, excludes manual dispatches,
and derives both the consecutive-failure count and newest failed gate from that
history. The live-collector regression supplies three consecutive failed push
runs, an older successful scheduled run, and an excluded failed manual run; the
old schedule-only query cannot satisfy the assertion.

The generated pipeline context named
`rust/crates/okp-core/src/playback_failure.rs` and
`rust/crates/okp-core/src/player_commands.rs`, but the complete successful
builder log and live health detail contain no playback diagnostic or player
command failure. Editing those portable domains would be unrelated to the
observed delivery-trigger defect.

## Limitations and holds

- The post-merge push-trigger and exact-main candidate promotion are proven.
  Fail-safe handling of repeated push failures is fixture-backed because
  deliberately breaking the public Windows delivery lane would be destructive.
- Rolling manifest and feed URLs intentionally advance. The exact build-33
  bytes are preserved in the issue comment and bound by checksums; Full packages
  use versioned asset names and digests.
- Hosted Windows Actions prove compilation, unit behavior, real-libmpv
  integration behavior, package assembly, and rolling-feed identity. They do
  not prove installation, updater migration, GPU decode, HDR, 4K60, physical
  audio, multi-monitor behavior, or any live Windows desktop state.
- The full live checker also reported an unrelated Linux candidate failure.
  This repair neither changes nor claims acceptance of the Linux delivery lane.
- No workflow was manually dispatched by this investigation, no recovery
  command was invoked or replaced, and no release, package, feed, tag, or
  repository setting was mutated by this investigation.
