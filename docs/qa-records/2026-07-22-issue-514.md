## Root cause and repair accounting

The alerted source was committed at `2026-07-21T20:06:54Z`. Its CI run was
created at `20:06:58Z`, and its Rust run was created at `20:06:59Z`. The first
gate failure was sampled only 7 seconds after the commit and 2–3 seconds after
workflow creation, exactly when the source jobs began. The second sample was
77 seconds after the commit; the CI integration job and Rust workspace job were
still running. CI completed successfully at `20:11:06Z`, and Rust completed
successfully at `20:09:40Z`.

Pull request #512 had merged the basic bounded settling behavior immediately
before these samples. Pending or not-yet-indexed exact-SHA CI evidence inside
the 900-second window became a blocking warning with reason
`source-main-ci-settling`; completed failures remained immediate failures, and
pending evidence remained fail-closed after the grace expired. Because the
alerted SHA is the repair merge itself, these samples also form the deployment
tail of the prior evaluator behavior: repository code in a newly merged source
cannot change a scheduler invocation that already began from an older checkout.

Pull request #524 completed the repair by anchoring settling to the matching
GitHub `PushEvent`, falling back to matching workflow creation time, and
treating a successful empty workflow query as pending rather than an API
failure. The evaluator and live collector retain regression coverage for
settling, overdue pending evidence, completed failures, delayed pushes, empty
run histories, and missing event-feed evidence. After #524 landed, Maestro
recorded that the configured runtime outcome was verified in the closure of
[`#508`](https://github.com/BeFeast/ok-player/issues/508#issuecomment-5039482974).

The generated pipeline context named
`rust/crates/okp-core/src/playback_failure.rs`, but the complete exact-source
workflow evidence contains no playback diagnostic failure. Editing playback
failure classification would therefore be unrelated to the observed gate
defect.
