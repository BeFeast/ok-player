## Root cause and repair accounting

Windows Candidate run `29882051896` started at `2026-07-22T01:03:53Z`,
completed successfully at `01:08:18Z`, and promoted build `16` from exact
source `1891d039eb00e2bd242c1cb6abc57c7e3e33b99d`. Its checkout, C# build,
unit tests, real-libmpv integration tests, package assembly, identity
validation, head revalidation, and feed-last promotion steps all passed. The
public workflow history therefore does not support an actual builder-failure
streak at the alert time.

`main` first advanced beyond that promoted source at
`2026-07-22T01:49:22Z`, with commit
`8e08f625cc51142aed0d0440f89476034c9303b2`. The first alert sample arrived
5,458 seconds after that commit, and the second arrived 5,546 seconds after it.
Both samples were below the versioned 7,200-second unpublished-main limit but
immediately beyond a 5,400-second boundary. This timing is consistent with a
deployed 5,400-second override or legacy threshold rather than the repository
contract, which deliberately includes a bounded 30-minute scheduler and
publication grace beyond the 60–90 minute delivery target.

GitHub did not create the next scheduled Windows Candidate run until
`2026-07-22T04:30:12Z`, more than three hours after the preceding scheduled
run despite the workflow's 15-minute cron expression. That run completed
successfully at `04:34:38Z`, promoted build `17` from the exact then-current
`main` source, and restored source equality. Every later scheduled Windows
Candidate run inspected through the current tested source also completed
successfully.

The external scheduler's invocation payload and environment overrides are not
public evidence, so this record does not claim which private runtime path
produced fingerprint `912f98e93df419c9`. It instead binds the public facts:
there was no consecutive builder failure, the samples were inside the
repository's 7,200-second contract, the next promotion passed, and the current
versioned checker reports the Windows row as passing. The full live checker
returned exit `1` only because of an unrelated Linux candidate row; the
sanitized issue evidence preserves that limitation rather than converting the
unrelated result into a project-wide pass.

The generated pipeline context named
`rust/crates/okp-core/src/playback_failure.rs`, but neither alert-adjacent
Windows Candidate log contains a playback diagnostic failure. Editing playback
failure classification would therefore be unrelated to the observed delivery
gate incident.
