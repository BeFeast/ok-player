## Root cause

`scripts/smoke-linux-idle-return.sh` closes media with `xdotool key --window "$window_id" --clearmodifiers x`. The `--window` form is an `XSendEvent` (`send_event=True`) that GTK4 does not route to the shortcut controller, so **Close Media never fires** — after a full run the isolated `XDG_STATE_HOME` holds no `history.json`, and `close-media-idle` is byte-identical to `eof-idle` (identical OSC residual `0.000095`, RMSE `0`).

Because Close never fires, the close-media step instead re-measures whatever is on screen ~4 s after the window maps (2 s activate + 2 s), which is exactly the 4 s fixture's natural EOF/clear boundary (~4.3 s in this environment). A capture that lands before the welcome is restored reads a structure-less surface (`identity_residual` 0.000–0.001) and fails the `identity_residual > 0.012` "Welcome identity" assertion — the failure the builder reported. Fast hosts clear before the capture and pass; slower builders do not. Throughout the entire lifecycle `alpha_min = 1`: the window is never transparent, so the `#467` opaque-idle contract holds and there is no ghost/transparent regression on this path.

A **real** Close (delivered via a focused XTEST event) exposes a second, independent defect in the step's expectation: closing a partially-watched file records resume progress (`save_current_progress(false)` → resumable entry), so the idle surface returns as the opaque **Continue-watching** welcome (parity with Windows `src/OkPlayer.App/Views/PlayerView.xaml`). That surface is opaque and OSC-free but is not pixel-identical to the empty first-run `initial-idle`; measured directly it sits at RMSE `0.153` vs initial. The step's `RMSE(close-media-idle, initial-idle) < 0.002` therefore encodes a surface the product does not render for a real close.

## Recommended harness fix (not applied here)

The gate assertion is the product contract and is owned by the smoke's author, so this record does not modify `scripts/smoke-linux-idle-return.sh`. The correct fix lives in the harness, in two parts, and strengthens rather than weakens the step:

1. Deliver the Close key as a real event so the step actually exercises Close Media, and assert it fired before capturing:

   ```sh
   xdotool windowfocus "$window_id"
   xdotool key --clearmodifiers x
   # Guard: fail loudly if Close did not fire, instead of silently racing EOF.
   for _ in $(seq 1 50); do
     [[ -f "$XDG_STATE_HOME/ok-player/history.json" ]] && break
     sleep 0.1
   done
   [[ -f "$XDG_STATE_HOME/ok-player/history.json" ]] || {
     echo "Close Media never fired (no history.json)" >&2; exit 1; }
   ```

2. Assert the real post-close **Continue-watching** contract — opaque (`alpha_min > 0.999`, no ghost), OSC absent, welcome identity present, video retired (`magenta_mean < 0.35`) — and drop `close-media-idle` from the pixel-identity-with-`initial-idle` (`RMSE < 0.002`) loop, which remains correct only for `eof-idle` (where `finished=true` retires the item to a neutral welcome).

## Limitations and holds

- `deb-idle-return-smoke` runs headless under Xvfb with `GtkGLArea` software rendering. It does not activate the native-Wayland `okp-native-video` / `has-active-video-plane` transparency path, so it can neither confirm nor clear the live GNOME/Wayland EOF transparency behaviour tracked by **#463**. That row is `NOT RUN` here and remains an operator/live-desktop hold.
- Candidate promotion past `deb-idle-return-smoke` stays **BLOCKED** until the harness race in the Close Media step is corrected. The fix belongs in `scripts/smoke-linux-idle-return.sh` (recommended above) and is reserved to the smoke's owner; no `okp-linux-gtk` regression was found to fix.
