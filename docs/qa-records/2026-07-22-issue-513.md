## Root cause and repair accounting

Issue #513 records two failures of the configured `healthcheck_command`. The
public escalation does not retain the private scheduler log or an alerted
source SHA. The immediately following repository incident, issue
[`#528`](https://github.com/BeFeast/ok-player/issues/528), identified the
concrete command defect: the live collector created a temporary snapshot
directory with an `EXIT` cleanup trap and then replaced itself with the Rust
evaluator via `exec`. Shell replacement bypassed the trap, so repeated ticks
abandoned snapshots until temporary storage was exhausted. PR #530 stopped
replacing the shell, added repeated cleanup coverage, and the operator then
recorded that Maestro had verified the configured runtime outcome.

PR #530's authoritative review also reproduced a lifecycle regression in that
repair: a supervisor sending `SIGTERM` only to the collector PID did not signal
the evaluator child. The evaluator could survive the supervised command and
delay scratch reclamation. Tested source
`bbc9c11c1859925a530054d92b97ea2469340602` keeps the collector alive, tracks
the evaluator PID, forwards `SIGTERM`, waits until the evaluator stops, returns
the conventional status `143`, and then lets the existing `EXIT` trap reclaim
the snapshot directory. Normal evaluator exit codes remain unchanged.

The regression test launches a controlled live collector, waits until its
evaluator is active, signals only the collector PID, and verifies all four
observable requirements: the evaluator receives `SIGTERM`, the evaluator PID
does not survive, the collector returns `143`, and the isolated scratch root is
empty. Existing coverage continues to verify successful and unhealthy evaluator
statuses plus repeated per-tick cleanup.

The pipeline context named
`rust/crates/okp-core/src/playback_failure.rs`, but neither the historical
command defect nor the repair path touches playback diagnostics. Editing that
module would be unrelated to the observed healthcheck lifecycle failure.
