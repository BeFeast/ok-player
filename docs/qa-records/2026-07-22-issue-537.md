## Root cause and repair accounting

The two external gate samples were one minute apart. At both sample times,
scheduled run `29915802794` was still building: it began at
`2026-07-22T11:28:00Z` and did not complete until
`2026-07-22T11:44:01Z`. The latest completed scheduled evidence available to
the evaluator was therefore run `29908612449`, followed by run `29899218063`.
Both runs failed `headless-launch-smoke` after the visible main window closed
but the application process did not exit. The workflow log emitted the
repository-owned marker `candidate build ... failed at gate
headless-launch-smoke`, so the evaluator correctly classified an explicit
candidate-build failure streak rather than a stale-schedule or feed-only
condition.

Issue #536 and pull request #538 identified the first shutdown defect, but the
post-merge candidate still reproduced the same gate failure. Issue #542 and
pull request #543 completed the repair by unmapping the window before deferred
shutdown, bounding native render-thread teardown, and exercising the real
window-close path in the smoke. The run that was active during the alert later
failed earlier at `deb-idle-return-smoke` with an EOF Welcome residual of zero;
that separate race was owned by issue #539 and stabilized by pull request #541
without weakening the Welcome or blank-frame thresholds.

Candidate run `29926924248` then built exact source
`b866e0b8cddc3d56cf326f9baa0cbac4b379e491`. Its unchanged
`deb-idle-return-smoke` passed, its `headless-launch-smoke` completed, the build
was declared promotable, and build 144 was published to the rolling candidate
channel. The public acceptance comment on issue #542 records the subsequent
read-only project outcome as `linux-candidate-delivery=pass` for the exact
published source.

The generated implementation hint named
`rust/crates/okp-core/src/playback_failure.rs`, but the complete failing logs
contain GTK shutdown and idle-return smoke failures, not libmpv diagnostic
classification errors. Editing playback-failure classification would therefore
be unrelated to the observed fingerprint.

## Limitations and holds

- The two Maestro evaluations are reconstructed from the canonical public
  issue timestamps and immutable public GitHub workflow chronology. The
  external scheduler's private invocation payload is unavailable and is not
  copied into this public repository.
- The rolling candidate pointer is mutable. Build 144's versioned packages,
  complete run, source SHA, and checksum manifest remain durable even after a
  later accepted candidate advances the pointer.
- Later `main` commits and later candidate runs can create a new delivery lag
  or a different gate incident. Those later states are not evidence that the
  repaired #537 streak persisted, and this record does not accept or repair
  them.
- Xvfb proves deterministic package rendering, window lifecycle, and process
  teardown for the recorded candidate workflow. It does not prove native GNOME
  Wayland compositor behavior, physical-GPU decode, portals, focus, chooser,
  drag/drop, or clipboard behavior.
- There is no remaining hold for the #537 fingerprint incident: the concrete
  shutdown and idle-return blockers were repaired, exact-source CI succeeded,
  build 144 passed and published, and the post-promotion outcome was recorded
  as `linux-candidate-delivery=pass`.
