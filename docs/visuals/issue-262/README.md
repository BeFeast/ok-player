# Linux floating volume acceptance evidence

Issue #262 evidence captured at the required 1120x680 viewport after the GTK exact-input
focus repair. Every image in the state matrix was regenerated by the focused volume
smoke from the repaired build.

The deterministic capture hook induces exact visual states only: rest is closed at
78%, focus is open at 78%, zero is 0%, normal is 64%, unity is 100%, boost is 124%,
and muted starts from 72% before toggling to zero so restore memory remains visible to
the model. The light, bright, and dark substrates are deterministic planes below real
loaded-media chrome; they isolate over-video legibility from Xvfb GL-frame variance.

## Real input verification

The focused smoke also launches without `OKP_VOLUME_PREVIEW` and drives actual GTK
input through X11: pointer hover, track click, rapid wheel events, Shift-wheel, Tab
focus traversal, arrow keys, decimal exact-value commits through both Return and blur,
outside-target focus, and the `M` shortcut. It asserts each level change in both the
persisted setting and the MPRIS value projected from mpv's observed playback snapshot.

- [Pointer-driven rest](interaction-rest.png)
- [Pointer-driven open capsule](interaction-open.png)
- [Recorded interaction assertions](interaction-results.txt)

The run proves rapid 1% wheel accumulation, 0.1% Shift fine-step, 1% arrow step,
0-130 clamping, mute restore, UI-to-mpv-to-poll synchronization, and zero changed
pixels in the downstream OSC row when the capsule opens. The Xvfb regression types one
changed decimal and applies it with Return, then types a different decimal and proves a
direct outside Play click commits on blur, retains focus, and lets the capsule collapse;
live one-click handoff remains part of operator QA.

## Limits

These captures and interactions prove deterministic GTK/X11 behavior under Xvfb with
the Cairo renderer. They do not prove installed GNOME/Wayland compositor presentation,
desktop animation smoothness, real monitor scaling, or pointer/focus behavior across a
live desktop session. Live GNOME acceptance remains pending and is still required before
the Draft/WIP PR can be marked ready.
