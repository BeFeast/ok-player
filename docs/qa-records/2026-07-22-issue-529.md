## Root cause and repair accounting

The streak crossed two rapid `main` updates rather than observing a completed
workflow failure twice. Source `8cf8ae0df27a8e18c4d078451c6cc084bea854f8`
was committed at `2026-07-21T21:58:35Z`, and its CI and Rust workflows were
created at `21:58:38Z`. The first gate sample arrived at `21:58:58Z`, only 23
seconds after the commit and 20 seconds after workflow creation. The CI
real-libmpv integration job and the Rust workspace job were still running; CI
eventually completed successfully at `22:01:12Z`, and Rust completed
successfully at `22:01:50Z`.

Pull request #524 then merged at `21:59:29Z` as source
`fa7869578ddcb1c114149a9f29619b80a11c8080`. Its CI and Rust workflows were
created at `21:59:32Z`. The second gate sample arrived at `21:59:44Z`, only 15
seconds after the merge and 12 seconds after workflow creation. Every active CI
job and the Rust workspace job were still running. CI completed successfully at
`22:02:09Z`, and Rust completed successfully at `22:01:43Z`.

Pull request #512 had already changed pending exact-SHA workflow evidence inside
a 900-second window from an immediate failure to a blocking warning with reason
`source-main-ci-settling`. It kept completed workflow failures immediate and
made overdue pending evidence fail closed. Pull request #524 completed the
repair by anchoring settling to the matching GitHub `PushEvent`, falling back
to matching workflow creation time, and treating a successful empty workflow
query as pending rather than an API failure. The evaluator and collector retain
regression coverage for settling, overdue pending evidence, completed failures,
delayed pushes, empty run histories, and missing event-feed evidence.

Because the hardened repair merged between the two samples and its own
workflows were only 12 seconds old at the second sample, the remaining alert is
deployment-tail traceability rather than evidence that the repair contract
failed. The external scheduler's private invocation payload is unavailable, so
this record does not claim whether the second sample ran an older checkout or
encountered the pre-cutover empty-index edge. Both paths are covered by #524,
and the live read-only checker at `2026-07-22T17:13:08Z` exited `0`, kept the
project healthy, and reported `source-main-ci` as `pass` for exact source
`994cf5b41b943c4e735b3f86c8da7e6892b19261`.

The generated pipeline context named
`rust/crates/okp-core/src/playback_failure.rs`, but none of the four complete
alerted-source workflow logs contains a playback diagnostic failure. Editing
playback failure classification would therefore be unrelated to the observed
gate defect.
