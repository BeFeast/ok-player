## Diagnosis and repair accounting

The last pre-fix `main` run was pulled directly from the stable workflow rather
than inferred from the issue description. Run `29874829548` built both lanes
from source `538fe76ea47a66fca8184e4dfa1074ad66471c59`. Before failing, it reported:

- bundled glibc exclusion `PASS` for both package lanes;
- dependency equivalence `PASS` for 163 dynamic ELF objects in each lane;
- build-marker `PASS` for both lanes; and
- Debian-testing `ldd` `PASS` for all 163 dynamic ELF objects in each lane.

It then failed at `sequence-1-appimage-narrow-width` with
`no disjoint seam left of the overflow entry (neighbour shares bounds):
maxima=0.949615`. This rules out the three suspected primary causes: the stable
lane was using the Debian 13 builder, the glibc-component exclusion was active,
and the verification floor matched. The immediate observed blocker was the
render smoke, not a glibc symbol-floor rejection.

Pull request #532 fixed the complete matrix rather than weakening the gate. It
gave repeated smokes a long-lived dark fixture and fresh XDG persistence,
selected deterministic package-bound software rendering where pixel evidence
is required, and handled the remaining cross-distro JPEG ABI mismatch in the
shared bundled-runtime policy. Debian's direct mpv dependency now travels as
`libokp-libjpeg.so.62`; the native Ubuntu candidate uses the same policy to
carry its ABI as `libokp-libjpeg.so.8`. Stable and candidate continue to call
the same runtime collector, policy, package builder, and portability verifier;
there is no lane-specific policy fork.

The merged pull request had one authoritative review finding: the helper
removed its disposable XDG directory before copying a failed attempt into the
evidence tree. This follow-up keeps that directory until after the failed
attempt is copied, uses the visible `xdg/` evidence name so the existing
workflow artifact glob retains it, removes it from successful output, and adds
regression coverage for both infrastructure-retry and product-assertion
failures.

## Limitations and holds

- This was deliberately a build-only stable run. It did not create or edit a
  GitHub Release, publish packages, or move either updater feed.
- Xvfb establishes deterministic package rendering and lifecycle assertions.
  It does not prove a live GNOME Wayland compositor, hardware acceleration,
  file/folder chooser, portal, drag/drop, clipboard, focus, or input behavior.
- Installed-package acceptance from a separate execution environment was not
  required by issue #495 and was not inferred from the build runner.
- The accepted package bytes remain bound to exact tested source
  `1891d039eb00e2bd242c1cb6abc57c7e3e33b99d`. The review follow-up changes only
  failed-attempt evidence retention and does not re-identify or republish those
  packages. The separate post-merge confirmation package set is bound to
  `8e08f625cc51142aed0d0440f89476034c9303b2` and likewise was not published.
- Same-source native candidate run `29881555381` passed the shared bundled-mpv
  and native-equivalence portability gates, then failed at its separate
  `headless-launch-smoke` because a visible process/window lifecycle did not
  clear. That later candidate-only gate is not a stable portability failure and
  is not accepted or repaired by this issue.
- No publication hold remains for the build-only acceptance defined by issue
  #495. Deliberate publication still requires the repository's separate
  candidate and live-acceptance inputs.
