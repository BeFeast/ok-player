## Root cause and repair accounting

The alerted source was committed at `2026-07-21T21:43:34Z`. Its CI run was
created at `21:43:37Z`, and its Rust run was created at `21:43:38Z`. The first
gate failure was sampled only 16 seconds after the commit and 12–13 seconds
after workflow creation. The second sample was 62 seconds after the commit and
58–59 seconds after workflow creation. At both samples, the CI real-libmpv
integration job and the Rust workspace job were still running. CI completed
successfully at `21:46:14Z`, and Rust completed successfully at `21:45:49Z`.

Pull request #512 had already changed pending exact-SHA workflow evidence inside
a 900-second window from an immediate failure to a blocking warning with reason
`source-main-ci-settling`. It kept completed failures immediate and made overdue
pending evidence fail closed. Its first implementation still anchored the
window to commit creation time and did not accept a successful empty workflow
query as pending evidence. Those gaps could misclassify a delayed push or a
newly pushed source whose workflow runs had not appeared in the API index.

Pull request #524 completed the repair by anchoring settling to the matching
GitHub `PushEvent`, falling back to matching workflow creation time, and
treating a successful empty workflow query as pending rather than an API
failure. It merged at `2026-07-21T21:59:29Z`, about 15 minutes after the #527
samples. The evaluator and live collector retain regression coverage for
settling, overdue pending evidence, completed failures, delayed pushes, empty
run histories, and missing event-feed evidence.

The repaired behavior was reproduced directly against tested source
`4ce772117a14d89e8d10c61215adba9b67974b1e`. At `2026-07-22T16:38:09Z`, with
its exact CI and Rust workflows still active, the checker exited `0`, kept the
project healthy, and reported `source-main-ci-settling` 129 seconds into the
900-second grace. At `16:41:45Z`, after both workflows succeeded, the same
checker again exited `0` and reported `source-main-ci` as `pass`. The complete
sanitized outputs are retained in the [issue evidence comment](https://github.com/BeFeast/ok-player/issues/527#issuecomment-5048881571).

The generated pipeline context named
`rust/crates/okp-core/src/playback_failure.rs`, but the complete exact-source
workflow evidence contains no playback diagnostic failure. Editing playback
failure classification would therefore be unrelated to the observed gate
defect.

The first local workspace-test invocation had one timing-sensitive failure in
`new_source_load_resets_global_subtitle_scale_before_dispatch` while its libmpv
event pump did not observe the requested scale inside the one-second deadline.
The exact test then passed five consecutive focused reruns, the complete
workspace rerun passed, and the exact tested-source Rust workflow passed the
same suite. This non-reproducing local anomaly is disclosed here because it
contradicted the initial gate run; it did not match the alert fingerprint or
the historical workflow evidence and required no source change for #527.
